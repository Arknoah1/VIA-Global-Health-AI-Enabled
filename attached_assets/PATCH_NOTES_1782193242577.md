# Patch notes — 5 priority fixes

Apply with: `git apply via-global-health-security-patches.diff` from the repo root, or copy the changes in by hand using the line references below. All five are small, isolated, and don't change any public API shape or DB schema.

I ran each touched file through an isolated `tsc` check; all five compile clean. (`routes.ts` throws unrelated `req.session` typing noise when checked outside the full project — that's because the ambient `express-session` module augmentation lives in `server/index.ts`, not because of anything in this patch. A full `npm run check` in your normal dev environment will confirm.)

---

## 1. `server/storage.ts` — `upsertShippingDeals` now atomic
**Finding #17.** Wrapped the delete+insert in `db.transaction()`, exactly matching the pattern already used a few lines above in `upsertLogisticsData()`. If the insert ever fails partway, the delete rolls back with it instead of leaving `shipping_deals` empty.

No behavior change on the success path — same rows in, same rows out.

## 2. `server/scraper.ts` — scraped SKU sanitized before touching the filesystem
**Finding #11.** Added `sanitizeForPath()`, an allow-list sanitizer (`[a-zA-Z0-9._-]` only, same idea as `enrich-product.ts`'s existing `sanitizeFilename()`) and applied it to `sku` at the top of `downloadDocument()` before it's used to build a directory path or the returned URL string. A page declaring `"sku": "../../../etc"` in its JSON-LD now resolves to a harmless `etc` directory name instead of escaping `DOCUMENTS_DIR`.

Normal SKUs (`VIA-12345`, `ABC-001`) pass through completely unchanged — the sanitizer is a no-op for anything that was already a safe SKU.

## 3. `server/routes.ts` — organisation-type lock enforced server-side
**Finding #1.** One-line guard added to the `contactData.organizationType` branch in `POST /api/quote-requests/:id/messages`: it now only writes the field `if (contactData.organizationType && !quoteRequest.organizationType)`, mirroring the check that already existed a few hundred lines later for the AI-extracted (`flags`) path. Once an organisation type is set on a quote request, no further request — direct API call or otherwise — can change it.

If you *do* need a legitimate way to correct a wrongly-entered org type post-hoc, that should be a separate `requireAdmin` endpoint, not something the public chat endpoint accepts implicitly — happy to add one if useful.

## 4. `server/seo.ts` — JSON-LD script injection closed
**Finding #12.** `JSON.stringify(meta.jsonLd)` output now has `<` escaped to `\u003c` before being embedded in the `<script type="application/ld+json">` tag. This is the standard "JSON-in-HTML" fix — it's semantically invisible to anything that parses the script as JSON (`\u003c` decodes back to `<` correctly) but means a product name/description containing `</script>` can no longer break out of the tag.

## 5. `server/shipping.ts` — DHL rate multiplier clamped
**Finding #18.** `overallRateMultiplierSuggestion` is now coerced to a number and clamped to `[0.90, 1.15]` — the same range the prompt already asks the model to stay within — in two places: once right after parsing the OpenAI response (so the *cached* value is always safe), and again where the cached value is read back in `generateShippingEstimate()` (so a value cached before this fix, or written by any future code path, still can't slip through). Anything missing or non-numeric falls back to a neutral `1.0` rather than `NaN`/`undefined` propagating into a price.
